# Архитектура cc-autocraft

## Обзор
«Центральный сервер (Core) — распределённые воркеры (черепахи)».

### Core (компьютер)
- Управляет складом через проводную сеть.
- Ведёт базу рецептов (`data/recipes.dat`).
- Планирует дерево крафта и резервирует сырьё.
- Диспетчеризирует задачи свободным воркерам.
- Хранит выбранный сундук-сканер (`data/scanner.dat`).
- Рисует UI на мониторе.

### Workers (черепахи)
- Исполняют конкретные задачи крафта.
- Общаются с Core по Rednet (протокол `cc_autocraft`, v1.0).
- Работают через 2 соседних сундука: входной и выходной буферы.

### Сундук-сканер
Отдельный сундук для записи рецептов. Исключается из склада. В его центральной
сетке 3×3 (слоты 4,5,6 / 13,14,15 / 22,23,24) выкладывают крафт; результат
тест-крафта кладётся в слот 16. Левый блок 3×3 автоматически сдвигается в центр
(имитация верстака).

## Определение инвентарей (важно)
Раньше использовалось `peripheral.getType(name) == "inventory"`, что НЕ работает
для сундуков (их тип — `"chest"`), а fallback `peripheral.hasType` отсутствует в
старых CC:T и вызывает ошибку «call nil». Теперь инвентарь определяется по
наличию методов `list` и `size` (`util.isInventory`) — работает на всех версиях.
Это было корневой причиной «нечитаемого склада».

## Резервирование ресурсов
Сырьё (предметы, которые берутся напрямую со склада) бронируется в момент
планирования заказа. Промежуточные предметы (которые крафтятся) не
резервируются — они производятся задачами. Если сырья не хватает, заказ не
стартует (с откатом резервов).

## Порядок задач
Внутри одного заказа задачи идут «снизу вверх»: сначала сырьё/промежуточные,
потом итоговые. `dispatcher.isReady` не даёт родительской задаче стартовать,
пока не завершатся дочерние (`plan_id` + `order`). Независимые заказы могут
выполняться параллельно на разных воркерах.

## Выход рецепта (output_count)
Некоторые рецепты дают >1 предмета (ступени — 4, кнопки — 1, и т.д.). Количество
партий = `ceil(нужно / output_count)`. Это учитывается и в планировщике, и в
окне заказа UI («будет изготовлено: xN»).

## Сетевой протокол
Rednet, JSON-подобные таблицы. `protocol = "cc_autocraft"`, `version = "1.0"`.
Типы: `DISCOVER`, `DISCOVER_ACK`, `CRAFT_REQUEST`, `TEST_CRAFT`, `RESULT`,
`HEARTBEAT`.

## Персистентность
- `data/recipes.dat` — база рецептов.
- `data/dispatcher.dat` — очередь и воркеры с буферами.
- `data/scanner.dat` — выбранный сундук-сканер.
- `latest.log` — журнал.

Состояние восстанавливается после перезагрузки.
